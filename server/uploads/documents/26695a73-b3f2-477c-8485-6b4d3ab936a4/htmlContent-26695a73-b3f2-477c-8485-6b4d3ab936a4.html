НАЗНАЧЕНИЕ ДОКУМЕНТА

В этом документе представлены бизнес-процессы, функциональные возможности и действия, необходимые для модернизации интерфейса модуля обработки писем. Включены подмодули "Внешняя корреспонденция" (разделы "Входящие письма" и "Исходящие письма") и "Внутренняя корреспонденция" (раздел "Почта"). Описаны ключевые компоненты и действия, обеспечивающие корректную работу и взаимодействие с пользователем, включая обработку писем.

Документ охватывает следующие аспекты:

Общие сведения: Общее описание модуля и его роли в рамках приложения.

Бизнес-процессы: Описание процессов, связанных с обработкой писем.

Функциональные требования: Перечень всех необходимых функций и возможностей, которые должен поддерживать интерфейс.

Интерфейс пользователя: Описание пользовательского интерфейса и взаимодействия с ним.

2. ПРОЕКТИРОВАНИЕ И ВНЕДРЕНИЕ НОВОГО ПРОЦЕССА ОБРАБОТКИ ПИСЕМ

Необходимо реализовать новый бизнес процесс обработки писем с учетом всех функциональных возможностей описанных в данном документе. Перечень и названия основных компонентов, их содержание и функциональность могут быть изменены и дополнены в процессе предварительного проектирования с согласованием исполнителя.

3. ИСХОДНЫЙ КОД ПРЕДВАРИТЕЛЬНОГО ИНТЕРФЕЙСА

В дополнение к технической спецификации прилагается исходный код предварительного интерфейса. Этот интерфейс поможет в проектировании дизайна, разработки функционала интерфейса и лучшем понимании технического задания.

Ссылка на исходный код (GitHub): https://github.com/AMIRTJK/Chat.git

Если в процессе тестирования интерфейса обнаружатся ошибки или баги, или если потребуется начать процесс с нуля, вы можете воспользоваться функцией очистки данных (см. Рисунок №1). На главном экране в заголовке чата есть иконка настройки

, которая открывает окно с опцией для очистки всех данных и сброса всего пройденного прогресса.

Иногда из-за особенностей библиотеки json-server запросы на обновление данных могут не полностью очищать данные, что может привести к возникновению багов. В таком случае в папке src/data есть файл db.temp.json, который содержит очищенные данные в формате JSON. Чтобы применить изменения, необходимо удалить файл db.json и заменить его на db.temp.json, предварительно переименовав последний файл с db.temp.json на db.json.

6. ОСНОВНАЯ ОБЛАСТЬ

Основная область интерфейса разделена на две части: левую и правую.

Левая часть экрана

В левой части экрана расположена панель, которая содержит функциональные кнопки для визирования. Эта панель позволяет пользователю эффективно управлять и организовывать процессы обработки писем и содержит следующие элементы:

Информация визирующего - в самом верху отображается информация о визирующем, включая имя, должность и аватарка.

Панель функциональных кнопок:

Исполнитель: Кнопка для выбора исполнителей, ответственных за выполнение задач. (см. Рисунок №5)

Виза: Кнопка для выбора визы. (см. Рисунок №5)

Срок: Кнопка для установки сроков выполнения письма. (см. Рисунок №5)

Статус: Кнопка для указания статуса письма. (см. Рисунок №5)

Блок "Вложенные документы" (см. Рисунок №5): Здесь отображается список вложенных документов. Пользователь может просмотреть вложенные документы, нажав на соответствующий документ.

Одной из ключевых функций данного блока является возможность добавления дополнительных документов в процессе работы с письмом. В связи с этим, в процессе разработки необходимо предоставить сотруднику канцелярии возможность загружать дополнительные вложения на любом этапе обработки письма.

При нажатии на файл откроется PDF Viewer, аналогичный дизайну, представленному на рисунке №6.

Рисунок №6. Интерфейс PDF Viewer

Шаг 2. Исполнение «LVL 1» - (y.khayrullo)

8. ОПИСАНИЕ ПРОЦЕССОВ ОБРАБОТКИ ПИСЬМА

При обработке письма в системе предусмотрено три области (далее "чаты"), где участники могут обмениваться сообщениями.

Первый чат: между визирующим (f.kahhorzoda) и исполнителем LVL 1 (y.khayrullo).

Второй чат: между исполнителями LVL 2 (a.ramazonovich и z.ismoil).

Третий чат: общий чат между исполнителями LVL 2 (a.ramazonovich) и исполнителями LVL 3 (b.ramazonovich).

Четвертого чата не будет, то есть исполнители LVL 3 не могут визировать, а могут лишь приглашать участников в третий чат.

Процесс создания чата и визирования

В модуле обработки письма, визирующий (f.kahhorzoda) нажимает кнопку "Создать чат" и переходит к обработки письма.

Визирование:

Выбор исполнителей.

Установка сроков выполнения.

Присвоение статуса.

После этого для каждого исполнителя появляется чат с визирующим для обмена сообщениями.

Исполнители LVL 1:

Например, заместители министра финансов (y.khayrullo) ставят свою визу, выбирая исполнителей LVL 2.

Исполнители LVL 2:

Например, начальники управления или директора (a.ramazonovich).

У них будет доступен внутренний чат для обмена сообщениями.

Исполнители LVL 2 могут создавать собственные чаты (вкладки) и ставить визу, приглашая участников.

Исполнители LVL 3:

Например, начальники отдела или специалисты (b.ramazonovich).

У них есть возможность создать и завершить заключение.

Готовое заключение переходит во второй и третий чат.
